# AI 开发与合并工作流

这份文档定义当前仓库的 AI 开发指导规范。目标不是替代 Git 平台权限，而是让人在本地协作、以及 AI agent 参与开发时，遵守同一套可执行流程。

## 分支策略

- `main` 是集成分支，不作为日常开发分支直接修改。
- 日常开发应在独立工作分支进行。
- 分支命名建议：
  - `feature/<topic>`：新功能
  - `fix/<topic>`：缺陷修复
  - `chore/<topic>`：文档、脚手架、维护项
- 如果当前任务已经在非 `main` 分支上进行，应继续在该分支完成，除非负责人明确要求切换分支。

## AI 开发流程

1. 先读代码再动手。
2. 先确认影响范围是 `server/`、`web-ui/` 还是两者都有。
3. 只修改完成任务所需的文件，不做顺手大改。
4. 优先延续现有实现风格，而不是引入另一套结构。
5. 完成后运行对应验证命令。
6. 输出结果时明确说明改了什么、验证了什么、还剩什么风险。

## 验证要求

### 前端改动

位于 `web-ui/` 的改动，至少应执行：

```bash
cd web-ui
npm run build
```

如果改动涉及 React 逻辑、TypeScript 代码或组件结构，补充执行：

```bash
cd web-ui
npm run lint
```

### 后端改动

位于 `server/` 的改动，应执行：

```bash
cd server
go test ./...
```

### 全栈改动

同时修改前后端时，前端和后端的验证都要跑。

## Review 关注点

- 变更是否严格对应当前需求，没有无关扩写。
- 是否影响房间状态、玩家状态、座位状态、行动流转等共享逻辑。
- 是否保留了用户已有改动，没有误覆盖脏工作区内容。
- 是否存在明显回归风险但没有说明。
- 是否有该补而未补的验证。

## 合并到 `main` 前的要求

- 代码先在工作分支完成。
- 通过对应验证后再进入 review。
- review 时优先看功能正确性、回归风险、状态一致性、缺失验证。
- 未经 review 或未说明验证状态，不应宣称“可以直接合并到 `main`”。

## 适用范围

这份规范主要用于：

- 当前仓库内的人机协作开发
- Codex / Trae / 其他 AI agent 的改码流程约束
- 对改动进行 code review 或交付说明时的统一口径
