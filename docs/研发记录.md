# 研发记录

## 2026-06-15

### 弃牌交互反馈不足

- 背景：当前项目在玩家执行弃牌操作后，界面缺少足够明确的即时反馈。
- 问题表现：
  - 玩家点击弃牌后，缺少明显的动画反馈，用户不容易第一时间确认操作是否已经生效。
  - 如果当前牌局节奏较快，其他玩家可能也难以快速识别某个座位已经弃牌。
- 影响：
  - 操作确认感偏弱，容易让用户误以为按钮未生效或网络有延迟。
  - 牌局信息传达不够直观，影响桌面状态的可读性和游戏沉浸感。
- 建议方向：
  - 为弃牌动作增加座位级别的视觉动画，例如头像置灰、手牌下沉/淡出、座位出现“已弃牌”状态过渡。
  - 增加明确文本提示或轻量提示，例如“玩家 X 已弃牌”。
  - 如果同时做动画与提示，应优先保证反馈节奏简洁，避免提示过多干扰牌局进行。
- 后续实现建议：
  - 先在前端确认当前弃牌状态由哪个字段驱动，并梳理状态变化时机。
  - 优先补充低成本、确定性强的视觉反馈，再决定是否增加全局提示文案。
  - 完成后补一次多人对局验证，确认弃牌反馈在自己视角和其他玩家视角都一致。

### 启动时的资料修复逻辑存在误删风险

- 背景：当前服务启动时会执行一次用户资料绑定修复，尝试清理重复或历史遗留的 profile 数据。
- 问题表现：
  - 现有修复逻辑会先读取 `users` 表中的 `username -> profile_id` 绑定，再按 `username` 去 `profiles` 表中查找同名资料。
  - 如果查到多个同名 profile，会把不是当前绑定 `profile_id` 的其他 profile 直接删除。
  - 这里把“登录用户名”和“玩家展示昵称”当成了同一个概念，但 profile 名称本身是允许修改的展示字段。
- 影响：
  - 只要有其他玩家资料的昵称碰巧与某个用户名相同，就存在被启动流程误删的风险。
  - 该问题不是界面层异常，而是不可逆的数据损坏风险。
- 建议方向：
  - 不要再基于昵称匹配做自动删除式修复。
  - 将修复策略改成显式绑定迁移，或仅记录冲突并停止自动处理，等待人工确认。
  - 区分登录标识和展示昵称，避免继续用名字推断 profile 归属。
- 后续实现建议：
  - 先审查 `repairUserProfileBindings()` 的设计假设，确认哪些历史数据确实需要迁移。
  - 如果需要保留自动修复，至少先改成“只标记冲突、不删除数据”。
  - 补一组包含“用户名与昵称重名但属于不同 profile”的回归测试。

### 启动恢复的筹码返还流程缺少幂等保护

- 背景：服务启动时会扫描 SQLite 中残留的房间与座位快照，把桌面剩余筹码返还给 profile，再删除房间记录。
- 问题表现：
  - 当前恢复流程是“遍历残留房间 -> 给每个玩家资料加回 `players.chips` -> 删除房间/玩家记录”。
  - “返还筹码”和“删除残留记录”之间没有事务或幂等标记保护。
  - 如果服务在筹码已返还、但房间记录尚未删除时再次异常退出，下次启动会重复扫描并再次返钱。
- 影响：
  - 会出现重复返还筹码，导致 profile 总筹码被无故放大。
  - 这是资金一致性问题，优先级高于一般 UI 体验问题。
- 建议方向：
  - 把恢复阶段的“读取残留快照、返还筹码、删除快照”封装成单个事务。
  - 或者为已结算残留记录增加明确的幂等状态，避免同一份残留数据被重复处理。
- 后续实现建议：
  - 先定义恢复流程的幂等模型，例如按 room/player 维度做一次性结算标记。
  - 补一组“恢复过程中崩溃后再次启动”的模拟测试，确保不会重复返还。

### 买入与补码的扣款和桌面快照落库不是原子操作

- 背景：创建房间、加入房间、补码都会涉及长期账户筹码扣减和桌面筹码增加。
- 问题表现：
  - 当前流程通常是先扣 `profiles.chips`，再修改内存中的玩家桌面筹码，最后在 `broadcastState()` 时把 room/player 快照写回 SQLite。
  - 扣款、桌面状态变更、快照落库三步之间没有统一事务。
  - 如果服务在扣款之后、快照落库之前退出，就会出现“账户已扣钱，但数据库里没有可恢复的桌面记录”。
- 影响：
  - 用户可能出现总筹码被扣减，但重新启动后又没有对应桌面筹码可返还的丢账问题。
  - 这是资金流闭环不完整的问题，直接影响真实可用性。
- 建议方向：
  - 将“扣 profile 筹码 + 写 room/player 快照”改成一个原子提交单元。
  - 至少要保证买入和补码在异常退出后仍然可恢复，不会出现只有一半成功的状态。
- 后续实现建议：
  - 梳理创建房间、加入房间、补码三条路径的状态转换顺序。
  - 如果暂时不重构整体事务层，先为关键路径增加补偿机制或崩溃恢复兜底。
  - 完成后补一次“扣款后异常退出、重启恢复”的端到端验证。

### 手动离房后的前端状态收口不完整

- 背景：当前支持玩家主动离开房间，并在服务端返还桌面剩余筹码。
- 问题表现：
  - 前端点击“离开”后，只会本地清空 `roomSessionRef` 并发送 `leave_room`。
  - 服务端会把玩家从房间移除并广播房间状态，但不会给离开的这个连接回一个明确的 `left_room` 或空状态消息。
  - 结果是离开者前端本地仍可能保留旧的 `state/chat/showdown`，界面看起来像还在原房间中。
- 影响：
  - 用户手动离房后，UI 容易停留在旧牌桌状态，继续操作时才收到报错，体验和状态一致性都不好。
  - 这会放大“是否真的离房成功”的认知负担。
- 建议方向：
  - 为 `leave_room` 增加明确的成功响应，例如 `left_room`。
  - 前端收到该响应后统一清空房间态、聊天记录和结算弹层。
- 后续实现建议：
  - 设计离房后的标准收口流程，包括状态清空、房间会话清理、必要的提示文案。
  - 补一次“主动离房后立即重新加入 / 创建新房”的交互验证。

### 离房后长期总筹码不会立即同步到前端 HUD

- 背景：服务端在玩家离房、被踢、断线超时清理等场景下，会把桌面剩余筹码返还给 profile。
- 问题表现：
  - 目前手动离房完成后，服务端没有主动回推最新 `profile`。
  - 前端在离房后也没有主动重新拉取 profile。
  - 因此 HUD 和个人资料卡中的“总筹码”会继续显示离房前的旧值。
- 影响：
  - 用户看到的总筹码与数据库实际值不一致，容易误判返钱是否成功。
  - 虽然数据本身可能已落库，但前端表现仍然像错账。
- 建议方向：
  - 在离房成功后，由服务端主动回推最新 profile，或由前端主动触发一次 `get_profile`。
  - 将离房、被踢、解散房间等会影响长期余额的场景统一做资料刷新。
- 后续实现建议：
  - 建立“哪些事件必须刷新 profile”的清单，不要只在登录和补码时更新资料。
  - 完成后补一次多场景验证，确认 HUD 总筹码和数据库长期保持一致。

### 战绩字段已展示到 UI，但后端尚未真正维护

- 背景：profile 结构和前端个人资料卡中已经存在 `handsPlayed`、`handsWon` 相关字段展示。
- 问题表现：
  - 数据库 schema 已有 `hands_played`、`hands_won` 字段。
  - 前端资料区也会展示战绩。
  - 但当前结算和发牌流程里没有看到任何地方实际更新这两个统计值。
- 影响：
  - UI 展示了看起来可信的战绩数据，但这些值实际上长期不会变化或不可信。
  - 容易让用户误以为系统已经支持完整战绩统计。
- 建议方向：
  - 如果短期内不准备实现正式战绩统计，应先去掉或弱化该展示。
  - 如果要保留展示，就需要在每手结束和赢家结算路径里正式维护统计字段。
- 后续实现建议：
  - 先定义“手数”和“胜场”的口径，例如单人吃池、多人平分、无效手牌如何记。
  - 补一组结算统计测试，避免后续出现统计口径漂移。
