# 移动端适配方案

## 背景

当前前端视觉和交互明显以桌面宽屏为中心构建，PC 上成立，但在手机端会出现可视区域被挤压、信息层级混乱、触控操作成本偏高的问题。当前移动端问题主要集中在以下组件：

- `web-ui/src/components/GameShell.tsx`
- `web-ui/src/components/GameShell.css`
- `web-ui/src/components/game/TableScene.tsx`
- `web-ui/src/components/game/game.css`
- `web-ui/src/components/game/ActionBar.tsx`
- `web-ui/src/components/game/SeatView.tsx`

这份文档的目标不是泛泛讨论“做响应式”，而是给出现有代码可直接落地的移动端改造方案，用于排期、实现和验收。

## 当前问题拆解

### 1. 顶部 HUD 仍然是桌面信息组织方式

`GameShell` 顶部同时展示 Logo、房间号、阶段、底池、手数、总筹码、连接状态、连接按钮、头像入口、房间按钮。当前虽然允许换行，但在手机宽度下会有几个结构性问题：

- 关键牌局信息和辅助信息混在同一层，用户第一眼不知道该看哪块
- `hudPill` 数量过多，换行后会明显吃掉垂直空间
- 右侧按钮组和头像入口会与统计信息竞争宽度

结果是进入房间后，手机屏幕上半部分被 HUD 明显占用，真正的牌桌反而被压缩。

### 2. 房间面板仍然是桌面侧栏思路

当前 `gameLayout.panelOpen` 会把页面切成“牌桌 + 右侧面板”，在桌面可用，但在手机上存在结构性问题：

- 打开面板后，牌桌主舞台会继续变窄
- 房间创建、加入、管理、日志都塞在一个高密度抽屉里，滚动和切换成本高
- 面板入口在右上角，更接近桌面操作习惯，不适合单手触达

移动端更适合底部抽屉、全屏面板或分步表单，而不是右侧常驻抽屉。

### 3. 牌桌舞台的高度预算不足

当前 `tableStage` 固定为 `16:9`，并使用环形座位布局。桌面上问题不大，但在手机竖屏下会出现：

- 牌桌高度受限，公共牌、底池、座位、头像、手牌被压到很近
- 上下两端座位容易贴边，状态气泡和手牌动画空间不足
- 牌桌为了维持比例而整体缩小后，名字、筹码、状态标签会明显变难读

这不是单纯缩小尺寸能解决的问题，需要重新分配小屏下的牌桌信息密度和高度预算。

### 4. 行动区偏桌面交互，不够触屏友好

`ActionBar` 当前横向排布以下元素：

- 弃牌
- 跟注 / 过牌
- 快捷下注按钮组
- 滑杆
- 下注 / 加注按钮

在手机端会遇到：

- 横向元素过多，容易换行成拥挤堆叠
- 快捷下注按钮和主操作按钮尺寸偏小
- 滑杆和数值显示在窄屏下可读性与可点性都下降
- 用户需要横向扫视多个区域才能完成一次操作

对于扑克牌局，轮到自己行动时必须尽量降低认知切换和误触风险。

### 5. 多个浮层状态彼此独立，移动端容易叠层失控

当前页面状态由多个独立布尔值控制，例如：

- `overlayOpen` 控制房间面板
- `profileOpen` 控制资料弹层
- `showSettlement` 控制结算面板

这在桌面上还能勉强接受，但在手机上会放大问题：

- 多个弹层可能同时争抢视口
- 关闭顺序不明确
- 回到主牌桌的收口路径不稳定

移动端不适合继续采用“每个浮层各管各的”方式。

### 6. 触屏入口覆盖不完整

方案不能只盯 HUD、抽屉和行动区，还要覆盖原本依赖 hover 的操作。例如当前 `SeatView` 里的“踢人”按钮在桌面 hover 可见，但手机触屏没有 hover 语义。只改布局，不改触发方式，体验仍然是不完整的。

### 7. 视口、工具栏、键盘与安全区属于一等约束

当前样式大量使用 `100vh`、固定底部面板和固定弹层。移动浏览器上这些都会受到以下因素影响：

- Safari / Chrome 地址栏伸缩
- 底部浏览器工具栏
- 输入法弹出
- 刘海屏 / 圆角屏安全区

如果不在第一阶段就处理这些约束，就算视觉看起来“像手机页面”，主路径仍然可能不可用。

## 适配目标

本次改造以“手机可用性优先，视觉一致性第二”为原则，优先保证以下主路径：

1. 登录并进入大厅
2. 创建房间 / 加入房间
3. 查看牌桌与玩家状态
4. 在自己的行动轮次完成操作
5. 管理个人资料、补码、离座、离房
6. 查看结算并回到牌桌

验收标准：

- 手机竖屏下首次进入页面时，无需缩放即可完成登录
- 在常见宽度 `375px` 到 `430px` 间，进入房间后核心牌局信息不溢出
- 行动按钮具备足够触控面积，关键操作不依赖精细拖拽
- 房间管理面板在小屏下不会继续挤压牌桌主体
- iPhone Safari 和 Android Chrome 下，底部操作区不会被系统工具栏或安全区遮住
- 横屏场景下不出现明显遮挡、错位或无法点击的区域

## 断点与布局总原则

### 断点建议

建议至少使用三档布局，而不是仅靠一个 `max-width` 媒体查询硬压缩：

- `>= 1024px`：桌面布局，保留“牌桌 + 右侧面板”思路
- `640px - 1023px`：平板 / 小屏桌面，允许抽屉化，HUD 两行
- `< 640px`：手机布局，切换为移动端专用结构

如果后续需要进一步压实，可再针对 `< 430px` 增加更激进的缩减策略。

### 手机布局总原则

手机端建议采用“上方精简状态 + 中间牌桌 + 底部操作 + 底部抽屉入口”的结构：

- 顶部只保留最核心状态，例如房间号、阶段、底池、连接状态
- 牌桌作为主视觉区，占据首屏主要高度
- 行动区贴近底部，保证单手操作优先
- 房间管理、消息、个人资料通过底部抽屉或全屏面板进入

这比在手机上继续模拟桌面双栏结构更稳妥。

## 状态模型与交互约束

### 统一移动端面板状态

建议不要继续让 `overlayOpen`、`profileOpen`、`showSettlement` 这类状态在移动端各自独立运行。移动端应引入一个统一的面板状态模型，例如：

- `none`
- `room_sheet`
- `profile_sheet`
- `settlement_fullscreen`
- `seat_actions`

建议规则：

- 同一时刻只允许一个移动端主面板打开
- 结算面板优先级最高，打开时应覆盖其他面板
- 打开新的主面板时，旧面板应自动关闭
- 返回键、遮罩点击、拖拽关闭都应收口到同一个状态出口

这样可以避免弹层叠层和关闭路径混乱。

### 触屏操作规则

建议明确以下触发方式：

- 头像入口点击打开个人资料面板
- 房间入口点击打开底部房间抽屉
- 座位操作不再依赖 hover
- 主持人管理玩家时，使用点击座位弹出操作菜单，或统一收口到管理面板

对于“踢人”这类高风险操作，应保留二次确认。

## 视口与安全区约束

这是第一阶段必须处理的内容，不属于后期打磨项。

建议约束：

- 优先使用动态视口单位，例如 `100dvh`，并保留兼容兜底
- 所有底部固定区域都要考虑 `env(safe-area-inset-bottom)`
- 顶部固定区域考虑 `env(safe-area-inset-top)`
- 登录输入框、资料编辑、房间表单在键盘弹出时不能被完全遮挡
- 底部抽屉打开时，内部滚动与页面滚动要分离，避免滚动穿透

建议直接在方案层面规定：

- 任何固定在底部的行动区或抽屉都必须预留安全区 padding
- 任何使用 `100vh` 的核心容器都要复核是否应替换为动态视口方案

## 分模块方案

### 一、`GameShell` 重构方向

建议把当前 `gameHud` 拆成桌面版和移动版两种呈现，不一定拆文件，但至少要拆布局层级。

移动端建议保留：

- 房间号
- 当前阶段
- 底池
- 连接状态
- 头像入口
- 一个统一的“更多”入口

移动端建议下沉到二级面板：

- 手数
- 总筹码详细信息
- 连接按钮
- 房间创建 / 加入 / 日志切换

推荐结构：

- 顶部第一行：房间号、阶段、连接状态
- 顶部第二行：底池、当前用户摘要、更多入口
- 点击“更多”后打开底部抽屉，承载房间管理、日志、补码、离座、离房等操作

### 二、房间面板改成移动端底部抽屉

当前 `aside.roomOverlay` 更适合桌面。手机端建议切换为：

- 固定在底部的 `bottom sheet`
- 高度使用 `70dvh` 到 `85dvh`
- 顶部提供明确关闭柄和标题
- 内容内部滚动，不影响页面主滚动
- 底部带安全区留白

内容组织建议重排为三层：

- 第一层标签：房间、消息、管理
- 第二层：创建房间 / 加入房间切换
- 第三层：房间列表与具体表单

房间列表保留，但每项信息收敛到一到两行，避免移动端信息噪音过多。

### 三、`TableScene` 先做最小可用，再做视觉压实

这里不能整体后移到第二阶段。应拆成两部分：

第一阶段必须完成的最小可用项：

- 重新分配牌桌区域的高度预算
- 给公共牌区、底池区、座位最外层留出最小安全边距
- 降低手机端座位与状态标签的最小尺寸冲突
- 确保 6 人桌在常见手机竖屏下可玩

第二阶段再继续优化：

- 调整座位圆环参数
- 缩减装饰元素
- 优化名字、筹码、房主标识的视觉密度
- 调整公共牌、气泡、弃牌反馈的层级关系

约束建议：

- 本轮以 6 人桌手机竖屏可玩为最低目标
- 更高人数的手机布局优化可以放后续迭代，不在首轮阶段一强求一次性解决

### 四、`ActionBar` 改造成移动端优先操作区

这是移动端体验的核心改造点。手机端建议改为分层布局：

- 第一行：轮到你行动 / 当前下注 / 你的筹码摘要
- 第二行：`弃牌`、`过牌/跟注` 两个主操作按钮
- 第三行：快捷下注按钮组
- 第四行：滑杆或数字输入
- 第五行：当前下注值 + 主确认按钮

除布局之外，还需要补齐交互规则：

- 进入自己回合时，默认下注值如何初始化
- 街道切换或 `minRaise` 变化时，默认值是否重置
- 快捷按钮点击后，主确认区域如何同步反馈
- `All-in` 是否需要额外确认
- 当滑杆在小屏上表现不稳定时，是否提供数字输入兜底

建议明确：

- 手机端按钮高度不低于 `44px`
- 高频操作按钮优先用整块按钮，不依赖细小 pill
- 主确认按钮与当前数值不要挤在同一条过窄行内

### 五、`SeatView` 的触屏化

需要显式覆盖座位级触屏操作，而不是只讨论样式缩放。

建议：

- 移除仅靠 hover 暴露的移动端关键操作
- 主持人点击某个玩家座位后，再打开座位动作菜单
- 座位动作菜单内容控制在最小集合，例如查看信息、踢人
- 高风险操作继续要求二次确认

这样才能让“触屏体验优化”覆盖真实操作链路。

### 六、浮层统一移动端行为

建议为以下弹层定义统一规则：

- 个人资料：手机端改为底部弹层或全屏卡片，不再挂在头像右下角
- 结算面板：手机端允许接近全屏展示，保证牌型与分池信息可滚动查看
- 房间面板：统一为底部抽屉
- 座位动作：使用轻量动作表，不与结算面板并存

统一之后，状态切换会稳定得多，样式也更好维护。

## 推荐实施顺序

为了降低风险，建议分三个阶段推进，但要调整优先级。

### 阶段一：先保住手机主路径可用

目标是“真机上能顺畅玩”，不是只在 DevTools 看起来像手机。

建议改动：

- 引入动态视口与安全区处理
- 改造 `GameShell` 顶部 HUD 的信息层级
- 将房间面板切换为手机底部抽屉
- 建立统一移动端面板状态模型
- 调整 `ActionBar` 为触屏友好布局
- 完成 `TableScene` 的最小可用压缩与高度预算重分配

阶段一完成标准：

- 登录、进房、行动、离房、查看结算这几条主路径在手机真机可完成

### 阶段二：压实牌桌与触屏细节

建议改动：

- 调整 `TableScene` 的座位位置参数
- 缩减装饰、重设头像 / 昵称 / 筹码 pill 尺寸
- 完善 `SeatView` 的触屏操作菜单
- 优化公共牌、底池、行动气泡在小屏下的层级关系

阶段二主要解决“能用但还挤”和“能操作但不够顺手”的问题。

### 阶段三：统一体验与打磨

建议改动：

- 结算面板移动端排版
- 个人资料移动端编辑体验
- 横屏与旋转切换后的状态保持
- 触控反馈、滚动锁定、抽屉开合动画优化

这一阶段更偏一致性与完成度。

## 验证建议

### 设备覆盖

至少验证：

- iPhone Safari 竖屏
- Android Chrome 竖屏
- 手机横屏
- 浏览器 DevTools 设备模拟

建议不要只做模拟器验证，至少补 1 台 iPhone Safari 和 1 台 Android Chrome 真机验证。

### 关键场景

1. 未登录状态下完成登录
2. 创建房间并进入牌桌
3. 打开房间抽屉、查看日志、关闭抽屉
4. 自己轮到行动时完成弃牌、跟注、加注
5. 打开个人资料并修改昵称
6. 主持人执行一次玩家管理操作
7. 查看结算面板并返回牌桌

### 重点检查项

- 是否出现横向滚动
- 抽屉 / 弹层是否遮挡关键按钮
- 底部操作区是否被浏览器工具栏或安全区遮住
- 键盘弹出后，输入框与确认按钮是否仍可达
- 抽屉内部滚动是否会带动页面滚动
- 横竖屏切换后，面板状态和牌桌布局是否错位
- 点击热区是否足够，是否容易误触

## 风险与注意事项

- 当前样式集中在 `GameShell.css` 和 `game.css`，如果只叠加媒体查询，维护成本会迅速上升，必要时应接受局部结构重排
- 牌桌是强视觉页面，移动端不应机械照搬桌面布局，否则只会得到“缩小版桌面”
- 当前页面已有多个独立状态源，若不先统一状态模型，移动端弹层问题会反复出现
- 如果后续支持更多人数牌桌，手机端布局复杂度会继续上升，建议尽早把“6 人桌优先”定为首轮边界

## 建议产出

后续正式进入开发时，建议按以下产出推进：

1. 先出一轮低保真线框，明确手机端页面结构与面板流转
2. 按阶段一到阶段三逐步实施，不建议一次性推倒重做
3. 每阶段完成后运行前端构建，并补一轮真机验证记录

这样可以避免一次性大改样式后难以回收问题，也能让方案直接转成研发任务。
